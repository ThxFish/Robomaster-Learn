# 哨兵机器人电控固件项目说明
## 项目概述
本项目为**RoboMaster 机甲大师赛 哨兵机器人**专属底层电控固件，基于 `Meta-Embedded-NG` 框架个人学习分支开发，针对哨兵机器人控制逻辑深度特化。

硬件平台搭载**高主频 STM32（STM32H7 系列）**，依托 **FreeRTOS 实时操作系统** 实现多任务并发调度。

核心功能：接收遥控器/上位机（视觉、自动导航）指令，融合 IMU、电机编码器等车载传感器数据，通过控制算法闭环计算，精准驱动底盘、云台、发射机构等执行设备，完成哨兵机器人全自动/手动作战控制。

## 整体架构设计
项目采用**分层化、模块化、低耦合**设计，自上而下分为四层架构，层级职责清晰，依赖隔离，便于功能迭代与跨平台移植。

### 1. Application 应用逻辑层
目录路径：`application`
机器人核心业务逻辑层，基于**发布-订阅（Publish-Subscribe）**通信机制设计。
模块间无直接函数调用与强头文件依赖，统一通过 `message_center` 消息中心完成跨模块数据交互，实现完全解耦。

主要业务模块：
- **Cmd 控制中枢**
  对应文件：`auto_cmd.c` / `robot_cmd.c`
  负责解析手操遥控器、视觉上位机下发的作战指令，转换为云台角度、底盘运动、发射机构等控制目标值，通过话题（Topic）对外发布控制指令。

- **Gimbal 云台控制**
  对应文件：`2yaw_gimbal.c`
  订阅云台控制话题指令，融合姿态传感器数据，通过 PID 算法计算 Yaw/Pitch 轴电机输出力矩与电流，实现云台增稳、随动、瞄准控制。

- **Chassis 底盘控制**
  对应文件：`agv_chassis.c`
  订阅底盘运动控制指令，通过运动学逆解算法，分配各轮毂电机目标转速，实现全向移动、姿态保持。

- **Shoot 发射机构**
  对应文件：`auto_ammo_booster.c` / `shoot.c`
  管控拨弹盘、摩擦轮电机逻辑，实现单发、精准连发等发射模式控制。

### 2. Module 组件算法层
目录路径：`module`
承上启下中间层，向上为应用层提供通用工具与算法能力，向下封装外设、传感器、设备驱动。

核心内容：
1. **硬件设备封装**
BMI088 IMU 惯性传感器、大疆无刷电机通用驱动、遥控器数据解析、裁判系统（Referee）协议解析。
2. **通用核心组件**
`message_center` 消息中心：发布订阅通信中间件核心；
`algorithm` 算法库：集成 PID、卡尔曼滤波、姿态解算等通用控制算法。

### 3. BSP 板级支持包层
目录路径：`bsp`
STM32 底层外设二次封装隔离层，基于 HAL 库开发。

核心作用：
- 统一封装片上外设：CAN、PWM、SPI、USART 等硬件接口；
- 标准化外设初始化、设备注册、收发回调接口；
- 隔离 CubeMX 与底层硬件耦合，剥离业务逻辑与硬件底层代码，提升固件跨芯片、跨平台可移植性。

### 4. FreeRTOS 实时任务调度
项目基于 FreeRTOS 多任务管理，按**不同运行频率、实时性优先级**划分核心任务，保障电控闭环高频稳定运行。

1. **INStask｜惯性导航任务（1kHz 高频）**
高频读取 IMU 传感器数据，解算机器人 Pitch/Roll/Yaw 欧拉角姿态，为云台自稳、姿态控制提供核心感知数据。

2. **RobotTask / SentryTask｜业务逻辑任务（≥150Hz）**
运行哨兵核心上层逻辑，包含传感器状态检测、消息话题分发、多执行器闭环运算，统筹全机控制逻辑。

3. **MotorTask｜电机控制任务（200Hz~1000Hz）**
统一汇总所有电机控制指令，打包后通过 CAN 总线批量下发，保证动力控制的实时性与同步性。

4. **MonitorTask｜系统监控任务（低频）**
后台守护进程（Daemon），实时检测电机在线状态、CAN 总线负载、模块异常，规避设备掉线、总线拥堵引发的机器人失控问题。

## 整体运行流程

传感器原始数据采集 → 高频任务完成姿态、里程等数据解算 → 控制中枢解析上位机/遥控指令，制定运动与作战目标 → 消息中心话题广播目标指令 → 云台/底盘/发射机构订阅对应指令 → 结合传感器反馈做 PID 闭环计算 → BSP 层驱动 CAN 总线下发控制信号 → 执行电机动作，完成整机控制闭环。

本文档从**感知流（自底向上）**和**控制流（自顶向下）**两条完整数据链路，完整梳理 RM 机甲大师（哨兵/步兵）电控固件的**核心功能 + 实现原理**，是理解整个项目的总纲。

## 一、向上游走 · 感知流（自底向上 → 汇聚大脑）
**定义**：硬件传感器 → 底层驱动 → 模块解析 → 应用运算 → 状态回传
**作用**：把机器人“看到、感受到、运动到的真实世界”上报给控制中枢

### 1. STM32 硬件中断源头（数据入口）
所有外部设备数据统一由**硬件中断**捕获：
- CAN 设备（电机、电调、裁判系统）：`bsp_can.c:138`
  中断源：CAN Tx/Rx FIFO 中断
- 串口设备（视觉、上位机、遥控器）：`bsp_usart.c:67`
  中断源：DMA + IDLE 空闲中断

**作用**：不管是电机转子位置、编码器值、还是视觉下发的装甲板坐标，都在这里被硬件“第一时间接住”。

### 2. Module 层 · 原始数据解析与提取
BSP 收到数据后，立刻回调到设备解析函数，把原始数据变成可用值：
- 大疆电机解析：`dji_motor.c:162`
  处理：16 位编码器值、转子位置、实际电流、转速、防抖滤波、数据校准
- IMU 数据解析：`bmi088.c:88`
  处理：陀螺仪角速度、加速度计数据

**作用**：把机器语言 → 工程可用物理量。

### 3. Application 层 · 各子 App 闭环运算
以**云台、底盘**为核心，执行实时控制算法：
- 云台：`gimbal.c`
  输入：IMU 解算真实角度 + 遥控/上位机目标角度
  算法：`controller.c:88` **双环 PID + 梯形积分 + 微分先行**
  输出：目标角速度、电机输出电流

- 底盘同理，根据编码器与姿态解算做速度闭环

**作用**：计算“我现在在哪、差多少、要怎么动”。

### 4. Pub-Sub 回传状态 → 控制中枢
各模块通过消息中间件把自身状态**发布**出去：
- 中间件：`message_center.c:106-109`
- 发布话题：`gimbal_feed`、`chassis_feed` 等状态频道
- 订阅方：`auto_cmd.c` 控制中枢

**作用**：让大脑知道：云台角度、底盘速度、电机温度、发射状态等全车实时信息。

## 二、向下游走 · 决策控制流（自顶向下 → 执行动作）
**定义**：控制中枢 → 发布指令 → 模块订阅 → 运动分配 → 最高频下发
**作用**：大脑决策 → 转化为机器人真实动作

### 1. 控制中枢决策（大脑）
大脑任务：`auto_cmd.c:240`
运行频率：100Hz FreeRTOS 任务

**计算内容**：
- 遥控器摇杆值
- 视觉/导航上位机指令
- 全车状态反馈
- 云台底盘耦合/解耦逻辑
- 发射允许/禁止判断

最终输出：
- 底盘速度：Vx、Vy、ωz
- 云台目标角度
- 发射机构动作指令

### 2. 消息中心广播指令
中枢通过 **Pub 发布** 指令话题：
- 底盘指令：`chassis_cmd`
- 云台指令：`gimbal_cmd`
- 发射指令：`shoot_cmd`

各模块在初始化时已**Sub 订阅**对应频道：
- 底盘：`agv_chassis.c:112`
- 云台：`gimbal.c`
- 发射：`shoot.c`

**作用**：安全跨模块数据传输，无函数调用、无强耦合。

### 3. 末端执行器分配控制量
以底盘为例：
`agv_chassis.c` 接收宏观速度指令 Vx, Vy, ωz
→ 运动学正逆解
→ 分配给 4 个麦轮电机
→ 设置 `DJIMotorSetRef` → 写入 PID 目标值

云台同理：
目标角度 → 双环 PID → 输出电流目标值

### 4. 最高频电调输出（最终执行）
最高实时性任务：**MotorTask（最高 1000Hz）**

执行流程：
1. 统一缓存所有电机指令到 `sender_assignment`
2. 通过 `dji_motor.c:13` 统一打包
3. 经由 CAN 总线**广播下发**给所有电调

**作用**：保证电机控制最低延迟、最高同步、最稳实时性。

# 三、一句话总结整个电控逻辑
### 感知流（自底向上）
硬件中断捕获原始数据 → Module 层解析校准 → App 层 PID 闭环运算 → 消息中心发布状态 → 上报控制中枢

### 控制流（自顶向下）
控制中枢融合遥控/上位机信息做决策 → 消息中心发布指令 → 各执行模块订阅接收 → 运动学/算法分配 → 最高频 CAN 下发 → 电机执行动作
